# Design: `shield:writing-style` authoring-time guide

**Date:** 2026-06-01
**Status:** Approved (design); pending implementation plan
**Author:** ashwinimanoj

## Problem

Docs generated by Shield (`research`, `prd`, `plan`/TRD, `lld`) have no shared
writing-style discipline. Clarity guidance today is scattered across review
prompts (`problem-clarity`), a PM review dimension (`stakeholder-communicability`),
and the `summarize` skill — none of which enforce simple, clear, concise prose at
authoring time. Output quality varies by run and by author. There is no single
source of truth for "what good Shield prose looks like," and nothing ensures
future doc-authoring skills inherit that standard.

## Goal

Add one canonical writing-style guide that the doc-authoring skills apply while
writing, so Shield docs come out simple, clear, concise, and to the point — and
so future doc skills inherit the same standard by construction.

## Non-goals (v1)

- No PostToolUse lint hook (separate mechanism; can flag but not rewrite well).
- No new scored/blocking review-gate dimension in `prd-review`/`plan-review`.
- No rewriting of deterministically-rendered or structured content.

## Design

### 1. Canonical skill

- **Location:** `shield/skills/general/writing-style/SKILL.md` (the `general/`
  domain — applies across all doc types).
- **Invoked as:** `shield:writing-style`.
- **Frontmatter `description`** doubles as the auto-load trigger (skill-agnostic
  backstop): "Use when authoring or revising any Shield doc — research, PRD,
  TRD/plan, LLD — to make prose simple, clear, concise, and to the point."
- **Contents:**
  - Four enforced principles, each a concrete checkable rule with a
    before→after example:
    1. **Cut filler & redundancy** — no throat-clearing, hedging, restated points.
    2. **Plain language** — short sentences, active voice, common words; define
       unavoidable terms once.
    3. **Structure over prose** — tables/bullets over paragraphs; BLUF
       (bottom line up front).
    4. **Concrete & specific** — real users, numbers, files; no vague
       abstractions.
  - A **revision pass**: a short checklist the caller runs over prose before
    writing the doc out.
  - A **Common Mistakes** table.

### 2. Scope guard (critical)

The skill applies to **author-written prose only** — outcomes, summaries,
narrative sections, descriptions. It explicitly **excludes**:

- Marker-wrapped / deterministically-rendered regions (e.g. TRD §10 emitted by
  `render_trd_section.py`).
- JSON sidecars, schema-bound fields, stable anchors.

Reason: rewriting rendered content would trigger `validate_trd.py` drift errors
(`milestone_drift`, `unbounded_markers`). The skill states this plainly so it
never fights the renderers.

### 3. Wiring — current docs

Add one near-final **mandatory step** to each of the four authoring skills —
`prd-docs`, `plan-docs`, `lld-docs`, `research`:

> "Apply `shield:writing-style` to all author-written prose before writing the
> doc."

### 4. Contract — future docs inherit it

This is the durable answer to "how do future skills use it":

- **`CLAUDE.md`** — add a convention: any doc-authoring skill MUST apply
  `shield:writing-style` to its prose.
- **`updating-plugin-assets` skill** — add a checklist item so the rule surfaces
  exactly when someone creates or edits a doc-authoring skill (the skill
  auto-loads on plugin-asset edits).

### 5. Eval coverage (mandatory per CLAUDE.md)

- New eval under `shield/evals/writing-style/`:
  - **RED** fixture: a passage of bloated prose violating the four principles.
  - **GREEN** assertion: with the skill loaded, the output tightens the prose —
    cuts filler, uses active voice/plain language, prefers structure, and is
    concrete.
- Include the RED→GREEN paper trail in the PR body.

### 6. Version bump

Bump Shield's version in `.claude-plugin/marketplace.json`.

## How "future skills use it" works

`shield:writing-style` is the single source of truth (lever 1). The four current
skills consult it explicitly (wiring). Future doc skills inherit it via the
contract in `CLAUDE.md` + `updating-plugin-assets` (lever 2), which fires at the
moment a new skill is authored. The auto-load `description` is a best-effort
backstop (lever 3) for any path that misses the wiring.
