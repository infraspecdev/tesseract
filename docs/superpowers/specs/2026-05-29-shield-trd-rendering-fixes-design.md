# Shield TRD Rendering & Generation Fixes — Design

**Date:** 2026-05-29
**Status:** Approved for planning
**Scope:** `shield/` plugin (generators), with the bill-payments-platform TRD regenerated as the proof/eval.

## Problem

The TRD produced by Shield's `/plan` has five user-visible defects, surfaced via
`flow-research/docs/shield/bill-payments-platform-20260430/outputs/trd.html`:

1. **§10 Milestones render out of order** — `M1, M10, M11, … M16, M2, M3, …`.
2. **The plan.json link (and every other inline body link) is broken in the HTML.**
3. **§7 High-Level Design is ASCII art, not a rendered diagram, and is thin.**
4. **The TRD never shows how milestones relate to their detailed designs (LLDs).**
5. **No LLDs exist for the feature, and the TRD doesn't explain why.**

### Root causes (verified in this repo)

- **Ordering:** `shield/scripts/render_trd_section.py:62` sorts milestones with
  `sorted(milestones, key=lambda m: m["id"])` — a lexicographic string sort. The
  docstring itself warns "if a project ever grows to M10+, callers should
  normalise ids." This feature has 16 milestones, so the latent bug fired.
- **Broken links:** markdown at `{feature}/trd.md` is rendered by
  `shield/scripts/render-markdown.py` into `{feature}/outputs/trd.html`. `./`-relative
  body links (e.g. `./plan.json`) are emitted verbatim, so they resolve to
  `outputs/plan.json` → 404. The nav bar is correct only because the shell injects
  `../`-prefixed links; inline body links inherited from the markdown are off by one
  directory. This affects **every** relative inline link (PRD, plan.md, plan.json…).
- **§7 ASCII:** `/plan` emits the topology as ASCII inside a plain ` ``` ` fence.
  `render-markdown.py:41` already converts ` ```mermaid ` fences to
  `<pre class="mermaid">`, and the HTML shell already loads mermaid.js — so the
  plumbing works; only the generated content is ASCII.
- **No LLD links rendered:** the latest `/plan` (steps 12–13 of `commands/plan.md`)
  already derives `lld_components[]` and `milestones[].touches_lld[]` from each
  story's `design_refs[]` (where `doc == "lld"`) and drafts one
  `lld-<component>.md` per component. But **neither `render_trd_section.py` nor
  `validate_trd.py` references those fields**, and §13's LLD list is hand-written
  prose. So the data exists (free on regen) but is never rendered.
- **Why no LLD for bill-payments:** the TRD was generated by an older `/plan`
  (`v2.21.0`) that predated steps 12–13. Its `plan.json` has no `lld_components`,
  no `touches_lld`, and no draft files. Regenerating with today's `/plan` produces
  them. (LLD drafting happens at `/plan` time; `/implement` only *promotes* drafts
  to canonical `docs/lld/`.)

## Goals

- §10 milestones render in a sensible, deterministic, dependency-aware order.
- Every inline relative link works in the rendered HTML.
- §7 conveys structure and dynamics via real (mermaid) diagrams.
- The TRD visibly links each milestone to its detailed-design (LLD) component(s).
- The TRD honestly explains the LLD lifecycle and where each LLD lives.
- All future TRDs benefit (generator fix), proven by regenerating bill-payments.

## Non-Goals

- Hand-patching only the bill-payments output (rejected — fix the generators).
- Embedding per-milestone *architecture diagrams* in the TRD (rejected — the TRD
  is the "what fits where" level; per-component depth belongs in LLDs).
- Bulk-generating the 8 bill-payments LLDs by hand (they're drafted by `/plan` on
  regen and promoted at `/implement`; not part of this generator fix).
- Adding a new authored `milestones[].lld_refs` field (rejected — derive from the
  already-present `touches_lld[]`, which is itself derived from stories).

## Design

All changes live in `shield/`. Deterministic script changes (A, B, C) are the core;
D and E are generation-guidance changes verified through the regenerated artifact.

### A. Milestone ordering — `render_trd_section.py` + `validate_trd.py`

Replace the lexical sort with a **dependency-topological sort, ties broken by
numeric id**:

- Parse the integer from each id (`^M(\d+)$`).
- Topologically order so a milestone appears only after every id in its
  `depends_on`. Among milestones whose deps are all satisfied, pick the **lowest
  numeric id** (deterministic, stable).
- If `depends_on` references an unknown id or a cycle is detected, fall back to
  pure numeric ordering and do not crash (defensive; surfaces via tests).
- `validate_trd.py`'s drift check must use the **identical** ordering so the
  byte-range compare stays exact.
- Remove the stale "normalise ids for M10+" docstring caveat.

### B. Per-milestone LLD links + description — `render_trd_section.py`

Each milestone block gains:

- An optional **description** line rendered when a `description` (or equivalent
  prose) field is present in the milestone object; omitted otherwise.
- A **"Detailed design:"** line listing each component in `milestones[].touches_lld[]`
  linked to its canonical `docs/lld/<component>.md`.

Links are computed **purely from `plan.json`**, never from filesystem existence, so
the drift validator stays deterministic. They render as forward references whether
or not the draft/canonical file exists yet. When `touches_lld[]` is empty/absent,
the line is omitted.

The link path is computed relative to the TRD's location (feature folder) →
`../../lld/<component>.md` style, or kept as a repo-rooted path — chosen during
implementation to match how other TRD links are authored; the HTML relative-link
rewrite in C will keep it correct in the rendered output.

### C. Relative-link rewriting in HTML — `render-markdown.py`

Override markdown-it's `link_open` and `image` render rules to rewrite **relative**
hrefs/srcs by the directory delta between `--md` and `--out`:

- Compute `rel = relpath(dirname(md), dirname(out))` (e.g. `..` when md is in the
  feature folder and out is in `outputs/`).
- Prefix relative URLs with that delta and normalize.
- **Skip** absolute URLs (`http://`, `https://`, `//`, `mailto:`, etc.), root-relative
  (`/…`), and pure anchors (`#…`).

This is a general fix for all inline body links, not just plan.json.

### D. §7 multiple mermaid diagrams — generation guidance

Update the §7 authoring guidance (`/plan` command + the plan-docs skill /
`shield/schema/trd-sections.yaml`, wherever §7 shape is specified) so `/plan`
emits **three ` ```mermaid ` fences** instead of ASCII:

1. **Topology** flowchart — trunk, 7 ports, IN-BBPS + MVP-fallback adapters,
   event backbone, ledger.
2. **Sequence** diagram — quote → order → debit → fulfillment → partner-pending →
   sweeper recovery → settle.
3. **Residency / region-boundary** flowchart — `eu-west-2` / `me-south-1` /
   `ap-south-1` subgraphs with the IN-data boundary and cross-region links.

Renderer + mermaid.js already handle the fences; this is purely content.

### E. §13 LLD list driven by `lld_components[]` — generation guidance

- §13's "LLDs (to be authored)" bullet is generated from `lld_components[]` with
  consistent canonical links and an honest lifecycle note (drafted by `/plan`,
  promoted by `/implement`).
- Update the §10 "ID note" prose (currently *"The renderer orders entries lexically
  by sidecar ID"*) so it no longer contradicts the new ordering (A).

### F. Regenerate the bill-payments TRD (proof)

Re-run `/plan` against
`flow-research/docs/shield/bill-payments-platform-20260430/` so the output exhibits:
populated `lld_components[]`/`touches_lld[]`, drafted `lld-<component>.md` files,
mermaid §7, correctly-ordered §10 with LLD links, and working HTML links. This is
the concrete answer to "why no LLD."

## Evals (mandatory — CLAUDE.md)

- `test_render_trd_section.py`: topo+numeric ordering across M1…M16 (incl. a
  parallel/late-numbered milestone, an unknown-dep fallback, and a cycle fallback);
  per-milestone LLD-link rendering from `touches_lld[]`; description rendered when
  present and omitted when absent.
- `render-markdown` test: relative-link rewriting when `--md` and `--out` are in
  different directories; absolute/anchor/root-relative links left untouched.
- `validate_trd` drift test: drift stays green with the new §10 rendering (ordering
  + LLD links).
- §7 mermaid presence check: a fixture/regenerated §7 with ` ```mermaid ` fences
  renders to `<pre class="mermaid">` via `render-markdown.py`.

D and E are LLM-authored content; they are verified via the regenerated artifact
plus the lightweight mermaid/§13 presence checks above.
