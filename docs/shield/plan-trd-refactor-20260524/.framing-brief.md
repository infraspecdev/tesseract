# Research Framing Brief — `/plan` → TRD refactor

**Date:** 2026-05-24
**Source:** `shield:research-framer` dispatch on the topic "Recommended practices for HLD and LLD documents."

## PF1 — Stakeholders

- **Shield maintainer** — owns the `/plan` refactor; cares about format-drift and duplication risks.
- **Shield end-users (staff/senior engineers, tech leads)** — read the TRD as primary architecture artifact.
- **Shield end-users (junior/mid engineers)** — consume the plan via `/implement`; need stories to point unambiguously at HLD/LLD sections.
- **`/plan-review` reviewer agents** — consume the TRD; need predictable section anchors.
- **`/pm-sync`** — consumes `plan.json`; backward-compatible schema is a hard constraint.
- **`/implement`** — consumes stories + ACs; needs design refs to be resolvable.
- **Future LLD-authoring command** — input contract is set now.
- **PM-tool admins at adopter orgs** — work-breakdown must map cleanly to epic/story/task hierarchies.

## PF2 — Decisions to make

1. Canonical TRD section list (HLD + PM-lens milestones).
2. Canonical LLD section list + granularity choice.
3. Story-to-HLD/LLD reference mechanism (anchor IDs vs section numbers vs URLs vs new `plan.json` fields).
4. De-duplication contract across PRD ↔ TRD ↔ plan.
5. Drift-prevention strategy (schema validation, template fixtures, structural checks).

## PF5 — Success criteria

Research succeeds if at end of synthesis the requester can answer:
1. Canonical TRD section list with rationale, traceable to IEEE 1016 / Sommerville / Google-design-doc lineage.
2. Canonical LLD section list + defensible granularity choice.
3. Story-reference mechanism with trade-off table + backward-compat verdict for `/pm-sync`.
4. PRD ↔ TRD ↔ plan de-duplication contract stated as 3–5 rules.
5. Drift-prevention recipe with ≥ 2 mechanisms.
6. ≥ 3 named failure modes with countermeasures the TRD design can adopt.
7. PF7 voices each appear with a direct quote in the body.

## PF6 — Prioritized research questions

1. **(Highest)** What sections constitute a well-formed HLD vs. LLD per the canonical lineage?
2. What sections constitute a well-formed LLD, and at what granularity is it authored?
3. How do mature orgs link user stories to HLD/LLD sections?
4. How is the HLD-to-milestone translation done in practice?
5. How do mature orgs prevent content duplication across PRD ↔ TRD ↔ plan?
6. Known failure modes of HLD/LLD documents and countermeasures?

## PF3 — Scope boundaries

**In scope:** HLD/LLD doc conventions (post-2015), PRD/TRD/plan relationships, story-to-design linking mechanisms in Confluence/Notion/Jira/ClickUp, failure modes + countermeasures, IEEE 1016 lineage as definitional anchor, the reference Notion workspace's existing conventions, recent (last 3 years) practitioner discussion.

**Out of scope:** Regulated-industry design-doc requirements, UML notation guides, agile-vs-waterfall debates, AI-assisted-design-doc-generation meta-commentary, procurement cost analysis of Confluence vs Notion.

## PF4 — Timeline & constraints

**Timeline:** Active refactor; research consumed within days. Moderate depth, fast turnaround. 5–8 strong sources per stream.

- Eval-coverage mandate (CLAUDE.md).
