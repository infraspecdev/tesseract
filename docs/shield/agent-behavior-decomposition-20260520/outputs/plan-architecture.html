Agent Behavior Catalog Migration

Phase: Restructure PM + tech-lead reviewers around named-behavior catalog

Timeline: 3 PRs sequenced (PR 0 → PR 1 → PR 2) | Depends on: Research findings at docs/shield/agent-behavior-decomposition-20260520/research/1-behavior-decomposition/findings.md

Problem

Shield's PM reviewer agent (shield/agents/product-manager-reviewer.md) grades 9 of 13 PRD-review rubric dimensions in a single dispatch. The agent is asked to grade dims 1, 2, 3, 7, 8, 9, 10, 11, 12 all at once — producing visibly shallow output. The same anti-pattern affects the tech-lead reviewer (shield/agents/architecture-reviewer.md) at compressed scale: 2 dims × 6-7 eval points each, all graded in one shot.

The research surfaced a converging prescription from four independent traditions:

  • Atul Gawande / WHO Surgical Safety Checklist — 47% complication reduction switching from monolithic to named, sequenced checks.
  • Dan North / Liz Keogh (BDD) — "you can only describe so much behaviour in a single sentence"; naming bounds scope.
  • FLASK (ICLR 2024) — empirical: per-skill scoring beats skill-agnostic by ~4 Spearman points.
  • Schluntz & Zhang (Anthropic) — direct vendor endorsement: "For complex tasks with multiple considerations, LLMs generally perform better when each consideration is handled by a separate LLM call."

Key Insight: The community has not converged on per-file behavior decomposition for LLM agents — popular catalogs (VoltAgent 20.2k★) use flat inline bullets and produce shallow output; the highest-quality precedent (BMAD-METHOD 47.7k★) uses a single shared rubric file referenced by one validator agent. Shield is taking one step beyond BMAD: a flat catalog of one-file-per-behavior, with on-demand loading from each agent.md. This is novel for LLM agents but is a direct generalization of two established patterns — BMAD's shared-rubric and Shield's own skill→helper convention (SKILL.md references repo-scan.md).

Solution

Approach: A flat behavior catalog at shield/agents/behaviors/ with on-demand loading. Each agent.md gets a lean ## Behaviors section listing owned behavior file paths with the rubric dimension each contributes to. At dispatch time, the agent uses the Read tool to load each behavior file in sequence and applies it. Behavior files are agent-agnostic (frontmatter holds only name — no owned_by, no owned_dim); the agent.md is the authoritative index.

Key decisions and trade-offs

Decision 1: Flat catalog (not per-agent subdirs)

Chosen: shield/agents/behaviors/<name>.md — a single flat catalog.

Rejected: shield/agents/<agent-name>/behaviors/<behavior>.md — risks colliding with the Claude Code subagent loader which expects agents/<name>.md. Adjacent same-stem directories are ambiguous and can break agent discovery.

Trade-off: Flat catalog requires file naming discipline (kebab-case, no agent prefix in the filename). Mitigated by the verify-behavior-references.sh lint script (PR 0).

Decision 2: On-demand load via Read

Chosen: The agent.md instructs the agent to Read each behavior file when it reaches that step in the protocol. Token cost is pay-per-use.

Rejected: Inline ## Behaviors section in agent.md — would double the agent file size (~211 lines → ~400+ lines) and pre-load full behavior surface on every dispatch even when behaviors don't fire.

Trade-off: N+1 Reads per dispatch (9 for PM, 13 for tech-lead). Each behavior file is ~30 lines; total added context ~3-5k tokens per dispatch — well within Claude Code subagent budget.

Decision 3: Behavior frontmatter holds only name

Chosen: Minimalist frontmatter. No owned_by, no owned_dim. The agent.md is the authoritative index.

Rejected: {name, owned_by, owned_dim} in behavior frontmatter — creates two sources of truth that can drift, pre-commits a behavior to one owner, and blocks future cross-agent reuse without rename.

Trade-off: Readers of a behavior file in isolation don't know who owns it without checking agent.md. Mitigated by the verify-behavior-references.sh lint and by the file-naming convention (no agent prefix).

Decision 4: PR sequencing PM → tech-lead

Chosen: PR 1 = PM (larger surface area, 9 behaviors), PR 2 = tech-lead (13 behaviors).

Trade-off: PM is sequenced first for learning — structural issues surface earlier. But tech-lead may carry the larger user-visible depth gain because each tech-lead dim has 6-7 eval points being decomposed.

Decision 5: Merge gate = distinct-finding-count baseline

Chosen: Capture per-fixture baseline of distinct findings emitted by the PM agent on main. New envelope must produce >= baseline total, no fixture regresses by more than 10%.

Rejected: Manual qualitative review only — unfalsifiable; "or better" without a number doesn't gate anything.

Stretch goal: 1.3× baseline. Meeting stretch confirms decomposition produces deeper output, not just longer prose.

Deliverables

  • PR 0: shield/agents/behaviors/ directory + README.md convention doc + CLAUDE.md section + verify-behavior-references.sh lint + pre-commit hook
  • PR 1: Baseline JSON + 9 PM behavior files + restructured product-manager-reviewer.md + personas.md + scoring.md dual-envelope acceptance + PM1-PM11 → behavior mapper + CI gate
  • PR 2: 13 tech-lead behavior files + restructured architecture-reviewer.md + personas.md tech-lead dispatch + tech-lead baseline + CI gate extension

Rollback strategy

  • Per-PR revert: Each PR is independently revertible. Reverting PR 1 does not affect PR 2 (they touch different agents). Reverting PR 0 cascades — PR 1 and PR 2 depend on the catalog convention.
  • Dual-envelope transition window: scoring.md accepts both old ({dim_grades:{}}) and new ({behaviors:[]}) envelopes. Reverting a PM/tech-lead agent file is sufficient to fall back to the old shape without orchestrator changes.
  • Eval baselines preserved: baseline.json is committed to main pre-restructure and never overwritten retroactively — any future regression is provable against the pre-decomposition number.
  • CI gate is reversible: The check workflow file can be disabled if it produces noise during the transition; the underlying script remains for manual runs.
Out of scope for this phase: agile-coach (already behavior-shaped via AC1-AC12); DX engineer (anti-pattern flagger, different output shape); security / SRE / data-PM / design persona agents (future template consumers); rubric.md dimension-ownership changes; cross-plugin changes.
