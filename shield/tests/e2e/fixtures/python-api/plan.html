Execution Plan

API Hardening: Input Validation & 404 Error Handling

Project: python-api-example  ·  Generated: 2026-03-15 23:00 UTC  ·  2 epics · 6 stories · 14 tasks

E1
Input Validation via Pydantic Models
Replace raw dict parameters in task endpoints with typed Pydantic models to enforce field constraints and prevent mass-assignment vulnerabilities.
E1-S1
Enhance Task model with field constraints
Add Field() constraints to the existing Task model: min/max lengths, status enum, and extra='forbid'.

Acceptance Criteria

  • Task.title has min_length=1 and max_length=200
  • Task.description has max_length=1000
  • Task.status is constrained to 'open', 'in_progress', 'done'
  • Task model uses ConfigDict(extra='forbid') to reject unexpected fields
  • Sending an invalid status value returns HTTP 422 with a descriptive error
  • Sending an extra field (e.g. "admin": true) returns HTTP 422

Tasks

  • E1-S1-T1 Import Field and ConfigDict from pydantic
  • E1-S1-T2 Add Field(min_length=1, max_length=200) to Task.title
  • E1-S1-T3 Add Field(default="", max_length=1000) to Task.description
  • E1-S1-T4 Add Literal["open", "in_progress", "done"] type annotation to Task.status
  • E1-S1-T5 Add model_config = ConfigDict(extra="forbid") to Task
E1-S2
Use Task model in create_task endpoint
Replace the untyped task: dict parameter in POST /tasks/ with task: Task.

Acceptance Criteria

  • POST /tasks/ accepts only valid Task fields
  • Sending a request without title returns HTTP 422
  • Sending a title longer than 200 chars returns HTTP 422
  • Valid request returns the created task with id, title, description, status
  • The stored task preserves the validated values from the model

Tasks

  • E1-S2-T1 Import Task model into src/routes/tasks.py
  • E1-S2-T2 Change create_task signature from task: dict to task: Task
  • E1-S2-T3 Use task.model_dump() to build the stored dict instead of **task
E1-S3
Add TaskUpdate model and use in update_task endpoint
Create a TaskUpdate model with all-optional fields for partial updates; replace updates: dict in PUT endpoint.

Acceptance Criteria

  • TaskUpdate model exists in src/models.py with optional title, description, status fields
  • Each optional field carries the same constraints as Task (max_length, enum)
  • PUT /tasks/{task_id} uses TaskUpdate instead of dict
  • Sending an invalid status value in an update returns HTTP 422
  • Sending only a subset of fields performs a partial update (other fields unchanged)
  • Sending extra fields returns HTTP 422

Tasks

  • E1-S3-T1 Define TaskUpdate(BaseModel) in models.py with Optional fields
  • E1-S3-T2 Import TaskUpdate in tasks.py
  • E1-S3-T3 Change update_task signature from updates: dict to updates: TaskUpdate
  • E1-S3-T4 Use updates.model_dump(exclude_unset=True) for the partial merge
E2
404 Error Handling for Missing Tasks
All endpoints that look up a task by ID must return a structured HTTP 404 response when the task does not exist, rather than an unhandled KeyError.
E2-S1
Return 404 in get_task when task not found
Add existence check and HTTPException(404) to GET /tasks/{task_id}.

Acceptance Criteria

  • GET /tasks/nonexistent returns HTTP 404
  • Response body is {"detail": "Task not found"}
  • GET /tasks/{valid_id} still returns the task correctly
  • No unhandled KeyError is raised for missing IDs

Tasks

  • E2-S1-T1 Import HTTPException from fastapi in tasks.py
  • E2-S1-T2 Add if task_id not in tasks: raise HTTPException(404, "Task not found") to get_task
E2-S2
Return 404 in update_task when task not found
Add existence check and HTTPException(404) to PUT /tasks/{task_id}.

Acceptance Criteria

  • PUT /tasks/nonexistent returns HTTP 404
  • Response body is {"detail": "Task not found"}
  • PUT /tasks/{valid_id} with valid body still updates correctly
  • No unhandled KeyError is raised for missing IDs

Tasks

  • E2-S2-T1 Add if task_id not in tasks: raise HTTPException(404, "Task not found") to update_task
E2-S3
Return 404 in delete_task when task not found
Add existence check and HTTPException(404) to DELETE /tasks/{task_id}.

Acceptance Criteria

  • DELETE /tasks/nonexistent returns HTTP 404
  • Response body is {"detail": "Task not found"}
  • DELETE /tasks/{valid_id} still deletes and returns {"deleted": id}
  • No unhandled KeyError is raised for missing IDs

Tasks

  • E2-S3-T1 Add if task_id not in tasks: raise HTTPException(404, "Task not found") to delete_task
