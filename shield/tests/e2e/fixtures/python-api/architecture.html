Architecture Document

API Hardening: Input Validation & 404 Error Handling

Project: python-api-example  ·  Generated: 2026-03-15 23:00 UTC

1. Context & Problem Statement

The src/routes/tasks.py router currently accepts request bodies as untyped dict objects and performs no existence checks before accessing the in-memory store. This creates two classes of defects:

  • Injection surface — any arbitrary JSON key is persisted without validation (mass-assignment risk, unexpected types).
  • Unhandled runtime errors — accessing tasks[task_id] for a missing key raises an unhandled KeyError, causing a 500 Internal Server Error instead of a structured 404.

Both issues are straightforward to fix using FastAPI's built-in Pydantic integration and HTTPException. The existing Task model in src/models.py already provides the right foundation — it only needs field constraints added before being wired into the endpoints.

Current behaviour: GET /tasks/999 returns 500 Internal Server Error (unhandled KeyError) rather than 404 Not Found. Any client can also POST arbitrary fields like "admin": true which are silently stored.

2. Goals & Non-Goals

Goals

  • Enforce field-level validation on task creation and update via the Task Pydantic model.
  • Return HTTP 422 Unprocessable Entity (automatically by FastAPI) for invalid input.
  • Return HTTP 404 Not Found with a structured body for any endpoint that accesses a non-existent task.
  • Introduce a TaskUpdate model for partial updates (all fields optional).

Non-Goals

  • Persistent storage (still in-memory; a separate database migration story is not in scope).
  • Authentication / authorization (a separate epic; the security reviewer has already flagged this).
  • Pagination on list endpoint.
  • Custom 422 response shape (FastAPI's default shape is acceptable for now).

3. Architecture Decisions

ADR-1: Constrain the Task model rather than creating a new one

The existing Task model has the right fields (title, description, status). Rather than replacing it, we add Field() constraints in place and set model_config = ConfigDict(extra='forbid'). FastAPI's OpenAPI schema generation will automatically reflect the new constraints.

OptionProsCons
Extend existing Task ✓Minimal diff; reuses existing model; automatic OpenAPI updateExisting tests (if any) may need updating for max_length
New ValidatedTask modelNon-breaking for old codeDuplicate models, confusing naming, old model stays unvalidated

ADR-2: Separate TaskUpdate model for PUT endpoint

PUT /tasks/{task_id} is a partial update — not all fields are required. Rather than making Task fields optional (which would break the creation endpoint), we introduce a TaskUpdate model where every field is Optional[…] with the same constraints. model_dump(exclude_unset=True) is used to merge only the provided fields into the stored record.

ADR-3: HTTPException at the point of access, not middleware

Global error-handling middleware could catch KeyError and return 404, but this is fragile (it catches all KeyErrors, not just task lookups). Raising HTTPException(status_code=404) explicitly at each endpoint makes the intent clear, keeps each endpoint self-documenting, and plays well with FastAPI's automatic OpenAPI response-code generation. 

# Pattern applied to get_task, update_task, delete_task
if task_id not in tasks:
    raise HTTPException(status_code=404, detail="Task not found")

5. Before / After Comparison

src/models.py — before

from pydantic import BaseModel

class Task(BaseModel):
    title: str
    description: str = ""
    status: str = "open"

src/models.py — after

from typing import Literal, Optional
from pydantic import BaseModel, ConfigDict, Field

class Task(BaseModel):
    model_config = ConfigDict(extra="forbid")

    title: str = Field(min_length=1, max_length=200)
    description: str = Field(default="", max_length=1000)
    status: Literal["open", "in_progress", "done"] = "open"

class TaskUpdate(BaseModel):
    model_config = ConfigDict(extra="forbid")

    title: Optional[str] = Field(default=None, min_length=1, max_length=200)
    description: Optional[str] = Field(default=None, max_length=1000)
    status: Optional[Literal["open", "in_progress", "done"]] = None

src/routes/tasks.py — create_task (before → after)

# BEFORE
@router.post("/")
async def create_task(task: dict):
    ...
    tasks[task_id] = {**task, "id": task_id, "status": "open"}

# AFTER
@router.post("/")
async def create_task(task: Task):
    ...
    tasks[task_id] = {**task.model_dump(), "id": task_id}

src/routes/tasks.py — get_task (before → after)

# BEFORE
@router.get("/{task_id}")
async def get_task(task_id: str):
    return tasks[task_id]   # KeyError → 500

# AFTER
@router.get("/{task_id}")
async def get_task(task_id: str):
    if task_id not in tasks:
        raise HTTPException(status_code=404, detail="Task not found")
    return tasks[task_id]

7. Testing Strategy

All changes should be covered by pytest + httpx (via FastAPI's TestClient):

  • Happy path: valid POST/GET/PUT/DELETE requests still return 200/201.
  • 422 validation: missing title, oversized title, invalid status, extra fields.
  • 404 handling: GET/PUT/DELETE with a non-existent ID returns {"detail": "Task not found"}.
  • Partial update: PUT with only status does not overwrite title.
Outcome: After these changes the API will automatically validate all task input via FastAPI's Pydantic integration and return structured 404 responses for missing resources — both behaviours are observable in the auto-generated OpenAPI docs at /docs.
