Architecture: Input Validation & JWT Authentication

Plan: input-validation-and-auth · Date: 2026-03-20 · Project: python-api-example
security validation authentication fastapi

1. Context & Problem Statement

The FastAPI task management API currently has several critical gaps:

  • No input validation вҖ” route handlers accept raw dict parameters, allowing arbitrary data to be stored. No field length limits, no type checking, no enum validation on status fields.
  • No error handling вҖ” missing tasks raise unhandled KeyError exceptions that leak stack traces. All mutations return 200 OK regardless of operation semantics.
  • No authentication вҖ” all endpoints including create, update, and delete are publicly accessible with no identity verification.
  • Incomplete tests вҖ” only happy-path tests for health and create exist. No coverage for error cases, validation, or auth flows.

2. Goals & Non-Goals

Goals

  • Type-safe request/response models with field-level validation
  • Proper HTTP status codes (201, 204, 404, 422) and structured error responses
  • JWT-based authentication for mutation endpoints
  • Role-based access control (admin role for deletion)
  • Comprehensive test coverage for all new behavior

Non-Goals

  • Persistent storage (database) вҖ” remains in-memory
  • Production-grade user management вҖ” demo in-memory user store only
  • Rate limiting, CORS, or API versioning
  • Thread-safety for the global task counter

4. Key Decisions

ADR-1: Pydantic models over raw dicts

Decision: Replace all dict parameters with typed Pydantic models.

Rationale: FastAPI's native integration with Pydantic provides automatic request validation, OpenAPI schema generation, and type safety with zero additional framework code. Raw dicts allow injection of arbitrary keys and have no validation.

Trade-off: Slightly more upfront code, but eliminates an entire class of input validation bugs.

ADR-2: JWT with HS256 via PyJWT

Decision: Use PyJWT with HS256 symmetric signing for authentication tokens.

Rationale: HS256 is sufficient for a single-service API where the same service issues and verifies tokens. PyJWT is a lightweight, well-maintained library. RS256 would be appropriate for multi-service architectures but adds unnecessary complexity here.

Trade-off: Symmetric key must be kept secret on the server. Cannot verify tokens without the secret (no public key distribution).

ADR-3: FastAPI Depends() for auth injection

Decision: Use FastAPI's dependency injection system (Depends()) for authentication rather than middleware.

Rationale: Depends() is more granular вҖ” it can be applied per-endpoint, supports role-based checks, and integrates with OpenAPI's security scheme display. Middleware would apply to all routes and require path-based exemptions.

Trade-off: Each protected endpoint must explicitly declare the dependency (but this makes the security posture visible in code).

ADR-4: Public reads, authenticated writes

Decision: Keep GET endpoints public. Require authentication for POST/PUT/DELETE.

Rationale: Read-only access is low risk for a task list. Requiring auth for mutations prevents unauthorized data modification. Admin role required for deletion adds defense-in-depth.

6. Risks & Mitigations

Risk: In-memory user store is not production-ready.

Mitigation: Clearly document as demo-only. Design the auth module interface so the user store can be swapped for a database-backed implementation without changing the auth dependency chain.

Risk: JWT_SECRET must not be hardcoded or committed to source control.

Mitigation: Load from environment variable. Fail fast at startup if missing. Add .env.example with placeholder but ensure .env is gitignored.

Risk: Global task counter is not thread-safe under concurrent requests.

Mitigation: Out of scope for this plan. Document as a known limitation for future improvement (use UUID or atomic counter).
